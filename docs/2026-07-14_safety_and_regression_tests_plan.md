# 안전성(파괴 시도) 테스트 추가 및 테스트 디렉토리 재구성 (Plan)

Written at (KST): 2026-07-14 13:30

## 1. 작업 목적

지금까지의 테스트는 "정해진 입력을 정확히 처리하는가"(Correctness)에 집중되어 있었다.
이번 작업은 그보다 한 걸음 더 나아가, 아래 두 가지를 확인하려는 목적이다.

- **안전성(Safety) 검증**: 일부러 이상한 타입/값을 넣어 시스템을 "망가뜨려 보면서",
  실제로 무너지는 지점이 어디인지, 무너진다면 어떤 형태로 무너지는지(친절한 에러 vs
  이해하기 어려운 예외)를 확인한다.
- **회귀(Regression) 검증**: 이후 어떤 코드를 변경하더라도, 지금 확인된 정상/이상 입력
  동작이 그대로 유지되는지를 `npm test` 한 번으로 확인할 수 있는 테스트 스위트를 만든다.

추가로, 대화 중 "테스트 파일을 `tests/` 디렉토리에 모아 달라"는 요청이 있어 기존
소스 파일 옆에 두던(co-located) 테스트 파일들의 위치도 함께 재구성한다.

## 2. 현재 상태 (사용자/독자 관점)

- 테스트 파일이 `poc/*.test.js`, `app/*.test.js`처럼 소스 파일과 같은 디렉토리에
  섞여 있어, 폴더를 열었을 때 "구현 코드"와 "테스트 코드"가 한눈에 구분되지 않는다.
- 기존 35개 테스트는 정상 경로(happy path)와 일부 에러 케이스(빈 값, 없는 id, 깨진
  JSON 파일)만 다루고 있다. 예를 들어 아래와 같은 입력은 아직 어떤 테스트도 확인하지
  않고 있다.
  - 이름(name) 자리에 문자열이 아닌 값(숫자 등)을 넣으면?
  - 검색어(keyword) 자리에 문자열이 아닌 값을 넣으면?
  - 연락처 id를 문자열("1")로 찾으면 숫자 id(1)와 같은 것으로 취급될까?
  - 연락처 파일 내용이 배열이 아니라 객체/숫자/문자열이면?
  - JSON으로 저장하려는 값에 순환 참조가 있으면?
  - 정렬 대상 배열에 NaN이나 서로 다른 타입이 섞여 있으면?
- 즉, "정상적으로 잘 되는 것"은 확인되어 있지만 "이상하게 넣었을 때 어떤 일이
  벌어지는지"는 코드를 직접 읽어야만 알 수 있는 상태다.

## 3. 목표 상태

- 테스트 파일은 `tests/` 아래에 소스 구조를 그대로 미러링해서 모인다
  (`tests/poc/*.test.js`, `tests/app/*.test.js`). 구현 코드와 테스트 코드가
  디렉토리 수준에서 분리된다.
- 5개 대상 모듈(`json-poc`, `quicksort-poc`, `storage`, `contactStore`, `cli`) 전반에
  걸쳐, "이상한 타입/값"을 넣었을 때 실제로 어떤 결과가 나오는지(정상 처리/명확한
  에러/불친절한 예외)가 테스트로 문서화되어, 코드를 읽지 않아도 알 수 있다.
- 이렇게 추가된 테스트들이 기존 테스트와 함께 `npm test` 하나로 실행되며, 이 전체
  통과 여부가 앞으로의 모든 변경에 대한 회귀 기준선이 된다. 별도의 회귀 전용
  인프라(스냅샷 등)는 추가하지 않는다.

## 4. 대표 시나리오

| 시나리오 | 현재(실측) 동작 |
|---|---|
| `createContact([], { name: 123 })` | `TypeError: name.trim is not a function` (예상치 못한 예외) |
| `createContact([], { name: '   ' })` | `Error: name is required` (의도된 도메인 에러) |
| `search(contacts, 123)` | `TypeError: keyword.toLowerCase is not a function` |
| `findById(contacts, '1')` (실제 id는 숫자 1) | `undefined` (타입이 다르면 못 찾음, 크래시는 아님) |
| `deleteContact(contacts, '1')` (실제 id는 숫자 1) | `Error: Contact not found: id=1` |
| 연락처 파일 내용이 `{"a":1}`(배열 아님) | 검증 없이 그대로 반환 (`loadContacts`가 배열 여부를 확인하지 않음) |
| `saveJson`에 순환 참조 객체 전달 | `TypeError: Converting circular structure to JSON` (감싸지 않은 원본 예외) |
| `loadJson`에 0바이트 파일 전달 | `Invalid JSON in <경로>: ...` (파일 경로가 포함된 명확한 에러) |
| `sortArray([3, NaN, 1])` | `[1, 3, NaN]`, 길이 3 유지, 예외 없음 |
| `sortArray([3, '1', 2])` (타입 혼합) | `['1', 2, 3]`, 예외 없음 (결정적 결과) |

## 5. 제안하는 변경 방식

- (완료) 기존 5개 테스트 파일을 `tests/poc/`, `tests/app/`로 이동하고 상대 import
  경로만 수정한다. 이동 직후 `npm test`로 기존 35개 테스트가 그대로 통과하는지
  확인해 "재배치가 기존 동작을 깨지 않았음"을 회귀 검증한다.
- 각 테스트 파일에 "이상한 타입/값" 카테고리의 케이스를 추가한다. 정상 처리든
  예외든, **현재 실제 동작을 있는 그대로 단언(assert)** 한다 — 즉 이번 작업은
  버그를 고치는 작업이 아니라, 지금 동작을 테스트로 고정해 두는 작업이다.
- 스코프는 사용자가 선택한 범위로 한정한다: 전체 5개 모듈 / "이상한 타입·값" 계열
  입력. 대량 데이터·성능 한계, CLI 입력 오남용(빈 입력 반복 등) 시나리오는 이번
  범위에서 제외한다.

## 6. 예상 설명 구조 (result 문서)

- Before/After 표(현재 문서의 "대표 시나리오" 표를 그대로 재사용)
- 새로 추가된 테스트가 몇 개인지, 그중 "정상 처리를 확인한 것"과 "예외/한계를
  문서화한 것"을 구분해서 설명
- 발견 사항(Findings): 코드를 고치지 않고 넘어간 취약 지점 목록
  (예: `createContact`/`search`가 문자열이 아닌 입력에 대해 불친절한 `TypeError`를
  던지는 점, `loadContacts`가 파일 내용이 배열인지 검증하지 않는 점)

## 7. 모호한 점 / 가정 / 트레이드오프

- **테스트 대상 스코프**: "전체 모듈" + "이상한 타입/값"으로 확정 (사용자 확인 완료).
  대량 데이터/성능 한계, CLI 입력 오남용은 이번 범위에서 제외한다.
- **회귀 검증의 의미**: 별도 스냅샷/기준선 문서 대신 "기존 테스트 스위트 유지·강화 +
  `npm test` 전체 통과"로 확정 (사용자 확인 완료).
- **테스트에서 드러난 취약점을 고칠 것인가**: 이번 요청은 "테스트를 통해 안정성을
  검증하고 싶다"는 것이지 "고쳐 달라"는 것이 아니므로, 발견된 문제는 **고치지 않고
  Findings로만 보고**한다(전역 CLAUDE.md의 "스코프 밖 발견 사항은 보고만 하고
  손대지 않는다" 원칙과 동일한 기준 적용). 사용자가 필요하다고 판단하면 별도
  작업으로 진행한다.
- **테스트 디렉토리 구조**: `tests/` 아래에 소스 디렉토리 구조를 그대로 미러링하는
  방식을 선택했다(`tests/app/`, `tests/poc/`). 파일명은 기존 "대상 파일과 같은
  이름" 규칙을 그대로 유지하면서 위치만 옮기는 것이 가장 최소 변경이라고 판단했다.
  이에 따라 프로젝트 `CLAUDE.md`의 테스트 파일 규칙 문구도 함께 갱신한다.

## 8. 검증 방법

- 테스트 이동 직후 1차로 `npm test` 실행 → 기존 35개 테스트 전부 통과 확인(재배치
  자체에 대한 회귀 검증).
- 새 테스트 추가 후 2차로 `npm test` 실행 → 전체 테스트(기존 + 신규) 통과 확인.
  예외를 던지는 것이 "현재 동작"인 케이스는 `assert.throws`로 단언하므로, 이
  테스트들도 정상적으로 "통과"한다(코드 수정 없이도 Green 상태).
