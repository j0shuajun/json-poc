# Step 4. app/contactStore.js — 연락처 CRUD 순수 로직 (Plan)

Written at (KST): 2026-07-14 12:35

## 1. 작업 목적

"연락처를 어떻게 저장하는지"(`app/storage.js`, 파일 I/O)와 "연락처를 어떻게 만들고,
찾고, 고치고, 지우는지"(business logic)를 분리한다. 이 모듈은 파일이나 화면을
전혀 알지 못하는 순수 함수들로만 구성해, 이후 `app/cli.js` 없이도 로직만 따로
테스트할 수 있게 한다.

## 2. 현재 상태 (리더 관점)

- `app/storage.js`는 연락처 배열을 통째로 읽고 쓰는 것만 할 수 있다.
- "특정 연락처 하나를 추가/검색/수정/삭제"하는 로직이 아직 없다.

## 3. 목표 상태

- `createContact(contacts, fields)` — 이름/전화/이메일을 받아 새 연락처를 만들고,
  연락처가 추가된 새 배열과 생성된 연락처를 함께 돌려준다. id는 자동으로 매겨진다.
- `findById(contacts, id)` — id로 연락처 하나를 찾는다(없으면 `undefined`).
- `search(contacts, keyword)` — 이름/전화/이메일 중 하나라도 keyword를 포함하는
  연락처들을 찾는다.
- `updateContact(contacts, id, fields)` — 지정한 필드만 수정한 새 배열을 돌려준다.
  대상이 없으면 에러를 던진다.
- `deleteContact(contacts, id)` — 대상을 제외한 새 배열을 돌려준다. 삭제 전에
  대상이 실제로 존재하는지 먼저 확인하고, 없으면 에러를 던져 "안전한 삭제"를 보장한다.

## 4. 대표 시나리오

- `createContact([], { name: '홍길동', phone: '010-1234-5678', email: 'hong@example.com' })`
  → id 1이 자동으로 매겨진 연락처가 추가된 배열을 돌려받는다.
- `search(contacts, '홍')` → 이름에 '홍'이 들어간 연락처만 걸러진다.
- `updateContact(contacts, 1, { phone: '010-9999-0000' })` → id 1의 전화번호만 바뀐
  새 배열을 돌려받는다.
- `deleteContact(contacts, 999)`(존재하지 않는 id) → "연락처를 찾을 수 없다"는
  에러를 던지고, 원본 배열은 변경되지 않는다.

## 5. 변경 접근 방식

- 모든 함수는 `contacts` 배열을 입력으로 받아 새 배열/값을 돌려주는 순수 함수다
  (전역 상태나 파일 I/O를 직접 다루지 않는다 — 그건 `storage.js`의 책임).
- id는 `Math.max(...기존 id들) + 1`로 자동 채번한다(배열이 비어있으면 1부터 시작).
- `createContact`는 `name`이 비어있으면 에러를 던진다(연락처의 최소 식별 정보이므로).
  `phone`/`email`은 선택 항목으로 취급해 생략 시 빈 문자열로 채운다.
- `updateContact`/`deleteContact`는 대상 id가 없으면 즉시 에러를 던져, 존재하지
  않는 데이터를 조용히 무시하지 않는다.

## 6. 예상 설명 구조

- 별도 다이어그램 없이, Create/Read/Update/Delete 각각의 입력·출력 예시 중심으로
  result 문서를 작성한다.

## 7. 가정 및 리스크

- 가정: id는 화면에 노출되는 사용자 식별자이자 검색/수정/삭제 키다(자동 채번, 재사용 안 함).
- 가정: `search`는 대소문자를 구분하지 않는 부분 일치(substring) 검색으로 한다.
- 리스크: 낮음. 파일 I/O가 없는 순수 로직이라 테스트로 모든 경로를 검증하기 쉽다.

## 8. 검증 방법

- TDD로 진행: 생성/조회/검색/수정/삭제 각각의 정상 케이스와, 존재하지 않는 id에 대한
  수정/삭제 실패 케이스에 대한 테스트를 먼저 작성해 실패를 확인한 뒤, 최소 구현으로
  통과시킨다.
- `npm test`로 전체 테스트 통과를 확인한다.
