# Step 1. poc/json-poc.js — JSON 파싱/저장 PoC (Plan)

Written at (KST): 2026-07-14 11:40

## 1. 작업 목적

2단계(CRUD 앱)의 `app/storage.js`는 이 PoC의 코드 구조를 그대로 재사용할 예정이다.
따라서 이 단계에서 "JSON 파일을 읽고 쓰는 최소 단위 함수"의 형태와 예외 처리 방식을
먼저 확정해 둔다.

## 2. 현재 상태 (리더 관점)

- 아직 이 저장소에 JSON 파일을 읽거나 쓰는 코드가 전혀 없다.
- Node.js 표준 `JSON`/`fs` 객체를 어떻게 조합해야 "파일이 없을 때", "파일 내용이 깨졌을 때"
  안전하게 동작하는지 확인된 바 없다.

## 3. 목표 상태

- `saveJson(filePath, data)` — 데이터를 보기 좋게 들여쓰기된 JSON 문자열로 만들어 파일에 저장한다.
- `loadJson(filePath)` — 파일을 읽어 JSON으로 파싱한 결과를 돌려준다.
- 파일 내용이 올바른 JSON이 아닐 경우, 원인을 알 수 있는 명확한 에러 메시지와 함께 실패한다.
- 저장 후 다시 읽으면(round-trip) 원래 데이터와 동일한 값을 얻는다.

## 4. 대표 시나리오

- `saveJson('a.json', { name: '홍길동' })` 호출 후 `loadJson('a.json')`을 호출하면
  `{ name: '홍길동' }`을 그대로 돌려받는다.
- 파일 내용이 `{ invalid`처럼 깨진 JSON이면, `loadJson`은 "어느 파일의 JSON이 잘못됐는지"를
  포함한 에러 메시지와 함께 예외를 던진다.
- 존재하지 않는 경로를 `loadJson`에 넘기면, Node 표준 파일 시스템 예외(`ENOENT`)가 그대로
  전파되어 호출자가 "파일이 없다"는 사실을 구분할 수 있다.

## 5. 변경 접근 방식

- `poc/json-poc.js`에 `saveJson`, `loadJson` 두 함수만 둔다 (그 외 책임 없음).
- `saveJson`은 `JSON.stringify(data, null, 2)`로 들여쓰기를 적용해 사람이 읽기 쉬운 형태로 저장한다.
- `loadJson`은 `fs.readFileSync`로 읽은 뒤 `JSON.parse`를 `try/catch`로 감싸, 파싱 실패 시
  파일 경로가 포함된 에러로 다시 던진다(re-throw). 파일이 없는 경우는 별도로 감싸지 않고
  Node의 기본 `ENOENT` 예외를 그대로 전파한다 (원인이 이미 명확하기 때문).

## 6. 예상 설명 구조

- 별도 다이어그램 없이, "저장 → 읽기 → 예외 두 가지(깨진 JSON / 없는 파일)" 시나리오
  중심으로 result 문서를 작성한다.

## 7. 가정 및 리스크

- 가정: 이 PoC는 동기(sync) 파일 I/O(`readFileSync`/`writeFileSync`)를 사용한다.
  콘솔 CRUD 앱은 한 번에 한 명의 사용자가 순차적으로 조작하는 소규모 프로그램이라
  비동기 처리의 이점이 크지 않다고 판단했다.
- 리스크: 낮음. 새로 추가되는 독립적인 유틸리티 함수이며 기존 코드에 영향이 없다.

## 8. 검증 방법

- TDD로 진행: round-trip 저장/읽기, 들여쓰기 적용 여부, 깨진 JSON 예외 메시지, 없는 파일 예외
  전파에 대한 테스트를 먼저 작성해 실패를 확인한 뒤, 최소 구현으로 통과시킨다.
- `npm test`로 전체 테스트 통과를 확인한다.
