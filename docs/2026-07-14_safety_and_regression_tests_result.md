# 안전성(파괴 시도) 테스트 추가 및 테스트 디렉토리 재구성 (Result)

Written at (KST): 2026-07-14 14:10

## 1. 한눈에 보는 요약

- 테스트 파일 5개를 소스 코드 옆(`poc/`, `app/`)에서 `tests/` 디렉토리로 옮기고,
  구현 코드와 테스트 코드를 디렉토리 수준에서 분리했다.
- "이상한 타입/값"을 넣어 일부러 망가뜨려 보는 안전성 테스트 19개를 5개 모듈
  전체(`json-poc`, `quicksort-poc`, `storage`, `contactStore`, `cli`)에 추가했다.
- 코드는 고치지 않았다. 모든 새 테스트는 **현재 실제 동작을 있는 그대로 확인**하는
  용도이며, 그 결과 기존 35개 + 신규 19개 = 총 54개 테스트가 전부 통과한다.
- 이 54개 테스트 스위트가 앞으로의 모든 변경에 대한 회귀(Regression) 기준선이 된다.
  이후 어떤 코드를 바꾸든 `npm test` 한 번으로 "이전과 같은 방식으로 동작하는지"를
  확인할 수 있다.
- 테스트 과정에서 몇 가지 "이상한 입력에 불친절하게 반응하는 지점"을 발견했다.
  고치지는 않고 아래 7번 항목에 발견 사항(Findings)으로 남겨 둔다.

## 2. 왜 이 변경이 필요했나

기존 테스트는 "정상적으로 잘 되는 경우"만 확인하고 있어서, 사용자가 실수로 이상한
값을 넣었을 때(예: 이름 자리에 숫자, 검색어 자리에 숫자) 프로그램이 어떻게
반응하는지는 코드를 직접 읽어야만 알 수 있었다. 또한 테스트 파일이 소스 파일과
뒤섞여 있어 "지금 무엇이 테스트되고 있는지"를 폴더 구조만으로 파악하기 어려웠다.

## 3. Before / After

| 항목 | Before | After |
|---|---|---|
| 테스트 파일 위치 | `poc/*.test.js`, `app/*.test.js` (소스와 같은 폴더) | `tests/poc/*.test.js`, `tests/app/*.test.js` |
| 테스트 개수 | 35개 (정상 경로 + 일부 에러 케이스) | 54개 (기존 35개 + 이상 입력 19개) |
| "이상한 값을 넣으면 어떻게 되는지" | 코드를 읽어야 알 수 있음 | 테스트 이름과 결과로 바로 확인 가능 |
| 회귀 확인 방법 | `npm test` (범위가 좁음) | `npm test` (좁았던 안전 범위까지 포함해 더 넓게 커버) |

## 4. 대표 시나리오

아래는 이번에 추가된 테스트가 실제로 확인한 "망가뜨려 보기" 결과 중 일부다.

```text
# 이름 자리에 숫자를 넣으면
createContact([], { name: 123 })
→ TypeError: name.trim is not a function   (친절하지 않은 예외)

# 이름이 공백 문자만 있으면
createContact([], { name: '   ' })
→ Error: name is required                   (의도된 도메인 에러)

# id를 문자열 "1"로 찾으면 (실제 저장된 id는 숫자 1)
findById(contacts, '1')
→ undefined                                  (크래시는 아니지만 못 찾음)

# 연락처 파일 내용이 배열이 아니라 객체이면
loadContacts(filePath)  // 파일 내용: {"a":1}
→ { a: 1 }을 검증 없이 그대로 반환

# 순환 참조 객체를 저장하려 하면
saveJson(filePath, circularObject)
→ TypeError: Converting circular structure to JSON

# 정렬 대상에 NaN이나 다른 타입이 섞이면
sortArray([3, NaN, 1])  → [1, 3, NaN]        (예외 없이 개수는 유지)
sortArray([3, '1', 2])  → ['1', 2, 3]        (타입 혼합, 예외 없음)
```

## 5. 독자 입장에서 달라진 점

- `tests/` 폴더만 열어 보면 이 프로젝트의 모든 테스트를 한 번에 볼 수 있다.
- 각 테스트 이름이 "무엇을 이상하게 넣었을 때 어떤 일이 벌어지는지"를 문장으로
  설명하고 있어, 코드를 열지 않아도 동작 경계를 파악할 수 있다.
- 코드 동작 자체는 변경되지 않았다 — 이번 작업은 "현재 동작을 테스트로 고정"하는
  작업이다.

## 6. 영향 범위

- 실행되는 애플리케이션 동작에는 변화가 없다(코드 수정 없음).
- 테스트 파일 위치가 바뀌었으므로, 앞으로 새 테스트를 추가할 때는
  `tests/<대상 폴더>/<대상 파일>.test.js` 위치에 작성해야 한다(`CLAUDE.md` 갱신 완료).

## 7. 제약 / 후속 작업 / 발견 사항 (Findings)

이번 작업 범위는 "테스트로 확인"까지이며, 아래 발견 사항은 **고치지 않고 보고만
한다.** 필요하다고 판단되면 별도 작업으로 진행할 수 있다.

- `contactStore.createContact`/`search`는 `name`/`keyword`가 문자열이 아니면
  (예: 숫자) `TypeError: x.trim is not a function` 같은, 원인을 바로 알기 어려운
  예외를 던진다. CLI를 통한 입력은 항상 문자열이라 실제로는 발생하지 않지만,
  순수 함수 자체의 계약(contract)만 놓고 보면 취약한 지점이다.
- `storage.loadContacts`는 파일 내용이 배열인지 검증하지 않는다. 파일이 손상되어
  객체나 숫자가 들어 있어도 그대로 반환하며, 이후 `contactStore`의 배열 메서드
  호출 시점에서야 예외가 발생할 수 있다.
- `poc/json-poc.saveJson`은 순환 참조 객체를 감싸지 않은 원본 `TypeError`로
  그대로 던진다(같은 파일의 `loadJson`은 이미 파일 경로를 포함한 에러로 감싸고
  있는 것과 비대칭).
- 이번 범위는 사용자 확인에 따라 "이상한 타입/값"으로 한정했다. 대량 데이터/성능
  한계(예: 정렬 대상 배열의 재귀 깊이)와 CLI 입력 오남용 시나리오는 이번 범위에서
  제외했다.

## 8. Lessons

- "안전성 테스트"와 "버그 수정"은 서로 다른 작업이다. 이상한 입력을 테스트로
  확인하는 과정에서 실제로 취약한 지점이 여럿 드러났지만, 요청받은 것은 검증이지
  수정이 아니었으므로 고치지 않고 발견 사항으로만 남겼다. 이후 유사한 요청에서는
  "발견되면 바로 고칠지, 보고만 할지"를 미리 확인하면 더 매끄럽게 진행할 수 있다.
- 테스트 파일 위치처럼 프로젝트 `CLAUDE.md`에 이미 문서화된 관례를 사용자가 대화
  중에 직접 바꿔 달라고 요청하면, 코드 변경뿐 아니라 `CLAUDE.md` 해당 조항도 함께
  갱신해야 규칙과 실제 구조가 어긋나지 않는다.
